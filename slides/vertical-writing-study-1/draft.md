- それでも、まだ自然には選ばれていない

## スライドの流れ

### 1. 縦書きWebは、もうある。なのに、まだ珍しい

- pixiv小説
- カクヨム
- 教育系サービス
- 個人サイト
- でも、縦書きのWebサイトはまだ珍しい

言いたいのはこれだけ。

> 縦書きWebは、もうある。10年前から。
> なのに、縦書きのWebサイトはまだ珍しい。

`writing-mode` もある。
実例もある。
でも、縦書きのWebサイトはまだ珍しい。
問題は、存在しないことではなく、自然な選択肢になっていないこと。

ここでは「そんなの見たことない」と思わせないのが大事。
観客の頭の中に、pixiv小説でも、カクヨムでも、個人サイトでもいいから、
ひとつ具体物を立ち上げる。
それで十分。

### 2. 要件から実装までの大きな流れは、もうある

- W3C の i18n IG / WG
- Japanese Layout Task Force
- CSSWG
- WHATWG HTML
- ブラウザ実装

厳密な組織図を見せたいわけではない。
要件を集める場がある。
仕様へ渡す先がある。
実装で確かめ、また戻せる。
主戦場は、たぶん CSS 側にある。

ここで言いたいのは、
「誰も考えていないから進まない」のではない、ということ。
実需があるから、要件整理が要る。
要件整理があるから、仕様に渡る。
仕様が動くから、実装も動く。
考える人はいる。
仕様にする場もある。
実装する側もいる。
大きな流れ自体は、もうある。
でも、それだけでは自然には増えない。
そもそも Web の縦書きが一定は成立してきたからこそ、この流れ自体もここまで回っている。
そして、ここまで来たのも実需があったからだと思う。

- 一本目の本命は、[codex/4.md](/Users/berlysia/Desktop/縦書き_study/codex/4.md) の 10 枚目にあるループ図を前借りする案
- 実需 -> 要件整理 -> 仕様 -> 実装 -> 利用 -> 実需 の循環として見せる
- i18n WG / JLTF / CSSWG / WHATWG HTML は、その循環の途中にあるプレイヤーとして薄く置く
- 図の読み方は `利用者の実需が、要件整理と仕様と実装を回す` に寄せる

### 3. 流れはある。だが足りない

- JLReq は「何が必要か」を整理する文書
- 仕様本文そのものではない
- CSS は CSSWG、HTML は WHATWG HTML、最後は実装が受ける

流れがあることと、自然に使えることは別。
要件文書があることと、仕様や実装が十分であることも別。
だから「流れはある」で話を終えない。

> 仕様の遅さを JLReq のせいにはしない。

### 4. しかも、日本語だけの話ではない

- 中国語
- 韓国語
- モンゴル文字

書字方向の問題は、日本語ローカルの趣味ではない。
「縦書き対応」の一語では済まない。
次は、中国語・韓国語・モンゴル語まで射程を広げる。

この一枚で言いたいのは、
縦書きWebを「日本語の特殊な願望」に閉じないこと。
日本語の話として始めつつ、
実はもっと広い問題だと開く。
だから、この足りなさも日本語だけの話では終わらない。
