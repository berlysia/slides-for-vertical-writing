遊び①島を置く

横書きの流れの中に縦書きの島を置くと、視線の慣性が折れて、読みの時間密度が変わる。 まずはスクロールしながら下の記事を読んでみてほしい。

読む速さは、いつも同じではない

文章を読んでいるとき、視線はほとんど自動で走っている。 左から右へ、上から下へ。Webページの大半はこの慣性の上に設計されていて、 読み手がそれを意識することは少ない。

その流れの途中に、向きの違うテキストを置いてみる。

ひさかたの　光のどけき　春の日に
しづ心なく　花の散るらむ 紀友則『古今和歌集』

ここで読む速さが一度変わる。少なくとも私は変わる。 流し読みの慣性が切れて、その一角だけ時間の密度が上がる。 紙の雑誌では当たり前の技術で、見出しだけ縦に組む、引用だけ縦に置く、 という形で読みの速度はずっと操作されてきた。

もう少し長い文章で試す。行が増えると、島は横に育つ。

メロスは激怒した。必ず、かの邪智暴虐の王を除かなければならぬと決意した。 メロスには政治がわからぬ。メロスは、村の牧人である。 笛を吹き、羊と遊んで暮して来た。けれども邪悪に対しては、人一倍に敏感であった。 太宰治『走れメロス』

雑誌の一段落ぶんくらいの長さを、そのまま島にするとこうなる。

きょう未明メロスは村を出発し、野を越え山越え、十里はなれた此のシラクスの市にやって来た。メロスには父も、母も無い。女房も無い。十六の、内気な妹と二人暮しだ。この妹は、村の或る律気な一牧人を、近々、花婿として迎える事になっていた。結婚式も間近かなのである。メロスは、それゆえ、花嫁の衣裳やら祝宴の御馳走やらを買いに、はるばる市にやって来たのだ。先ず、その品々を買い集め、それから都の大路をぶらぶら歩いた。メロスには竹馬の友があった。セリヌンティウスである。今は此のシラクスの市で、石工をしている。その友を、これから訪ねてみるつもりなのだ。久しく逢わなかったのだから、訪ねて行くのが楽しみである。歩いているうちにメロスは、まちの様子を怪しく思った。ひっそりしている。もう既に日も落ちて、まちの暗いのは当りまえだが、けれども、なんだか、夜のせいばかりでは無く、市全体が、やけに寂しい。のんきなメロスも、だんだん不安になって来た。 太宰治『走れメロス』

横スクロールが生えた。行長を決めても、量が増えれば島は block 方向へ育ち続け、いずれ親の幅を超える。 しかも初期表示で見えているのは文章の末尾側で、書き出しは右の奥にある。 スクロールの開始位置はスクロールコンテナ自身の writing-mode で決まるためで、この枠は横書きのままにしてある。

枠にも縦書きを継がせると、書き出し側から始まる。

縦書きの本文を流すなら、この伸び続ける幅とどう付き合うかを先に決めることになる。 付き合い方の続きは遊び④で扱う。

これをWebでやってはいけない理由は特にない。プロパティは揃っているし、ブラウザは普通に組む。 やっている実例が少ないだけである。

縦書きでは寸法の役割が回転する。inline-size が行の長さ(=高さ)で、決めないと行長がビューポート任せになり、縦に間延びする。文章が増えたときに島が横へ育つのは block 方向の自然な成長で、こちらを抑えたいなら block-size や overflow の出番になる。

横書きの枠に縦書きをはみ出させると、スクロールの開始位置が左端=文章の末尾側になり、読み手は書き出しを探して右へ戻ることになる。はみ出しを受ける枠の writing-mode まで含めて決める必要がある。

遊び②引き算で確かめる

本当に効いているのかは、外して確かめるのが早い。切替は writing-mode 1プロパティで済む。 印象に効く変数はここでは2つ(縦書きと書体)あるので、別々に引けるようにした。

声に出して読む文章

声に出して読まれることを前提にした文章がある。 一文字ずつ降りていく縦書きは、こうした文章に合うように見える。 本当にそうかを確かめたい。

雨ニモマケズ
風ニモマケズ
雪ニモ夏ノ暑サニモマケヌ
丈夫ナカラダヲモチ 宮沢賢治「雨ニモマケズ」

横書きに戻しても、行長と中央寄せはそのままにしてある。動くのは進行方向だけで、 それでも読み味が変わるなら、その差は縦書きによるものだ。 残りの「特別さ」は明朝体の仕事かもしれない。 このため、書体だけを外すトグルも別に置いた。1変数ずつ引いて確かめられる。

切り替えると要素の高さが変わり、ページ全体がガタッと動く。実プロダクトで切替UIを出すなら、外側に min-block-size で場所を予約しておくと跳ねない。

ふりがなのために用意された仕様の密度について。

新着

行頭行末の約物

括弧と読点の、行頭・行末での扱いの話。

縦書きの中で数字は寝る(横倒しになる)。<span class="tcu">12</span> のように包んで text-combine-upright: all で縦中横にする。all は包んだ範囲の全文字を1文字ぶんの幅に詰め込む指定なので、何桁で使うかは書き手の責任になる(潰れずに収まるのは実用上2〜3桁まで)。桁数で自動制御する digits も仕様にはあるが、手元のChromiumは解さない。

遊び④段に流す(縦組みマルチカラム)

本文ごと縦書きにしたい。一方で、縦スクロールの世界は捨てたくない。 そのための接続が段組みだ。新聞がずっとやってきた形を、Webの縦スクロールに乗せる。

メロスは激怒した。必ず、かの邪智暴虐の王を除かなければならぬと決意した。メロスには政治がわからぬ。メロスは、村の牧人である。笛を吹き、羊と遊んで暮して来た。けれども邪悪に対しては、人一倍に敏感であった。きょう未明メロスは村を出発し、野を越え山越え、十里はなれた此のシラクスの市にやって来た。メロスには父も、母も無い。女房も無い。十六の、内気な妹と二人暮しだ。この妹は、村の或る律気な一牧人を、近々、花婿として迎える事になっていた。結婚式も間近かなのである。メロスは、それゆえ、花嫁の衣裳やら祝宴の御馳走やらを買いに、はるばる市にやって来たのだ。先ず、その品々を買い集め、それから都の大路をぶらぶら歩いた。メロスには竹馬の友があった。セリヌンティウスである。今は此のシラクスの市で、石工をしている。その友を、これから訪ねてみるつもりなのだ。久しく逢わなかったのだから、訪ねて行くのが楽しみである。

太宰治『走れメロス』

段の行長は column-width で自分から当てに行く必要があるが、指定した値がそのまま使われるとは限らない。仕様上この値は理想値の扱いで、実際の行長は空間を段数で割り直した長さに引き伸ばされる。実際、このデモの段も指定より長い行で組まれている(手元の計測では13em=208px指定に対して約240px)。コンテナの inline-size を段数×行長から逆算して固定すれば指定値は通るが、必要な段数は内容の行数と空間で変動するため、CSSだけで先に決めるのは実質成立しない。きっちり通したければ現状はJSで測って再計算するしかなく、このデモは「だいたい13em」で割り切っている(JSの話はJSConf JP 2025のスライド参照)。

そもそも、いまのマルチカラムには「与えられた空間を段に分ける」モデルしかない。「固定の段幅のまま、無限の空間に段をつらね続ける」という発想が仕様に存在しない。この欠落は CSSWG に問題として提起してある: csswg-drafts#13665 — Fixed column width mode for multi-column layout。起票したのは私で、困りごとはここまで持っていける、という実例として置いておく。

遊び⑤進行方向ごと明け渡す

ここまでは縦スクロールの中の遊びだった。最後は、ページの進行方向そのものを縦書きに渡す。 root に writing-mode: vertical-rl を置くと、ページは右から左へ進む横スクロールになる。 下の枠の中でホイールを回すと、そのまま左へ進む。

全画面で開く → 左下のHUDで scrollLeft が負に育っていくのを眺められる。

素朴に置いただけでは、Chromiumではマウスの縦ホイールでこのページを読み進められない。縦ホイールは縦スクロール専用にマップされていて、このページには縦スクロールがないため無反応になる。トラックパッドの横スワイプだけが効く。

スクロールの座標系も変わる。scrollLeft は初期位置(右端)が 0 で、読み進むほど負の値に育つ。スクロール位置を扱うJSは大体ここで一回壊れる。壊れた箇所はメモして持ち帰ってほしい。
